# 方案互审工作流

> 你（当前 AI）是**生成方**，负责产出项目开发方案。本 Skill 帮你调用 Claude Code 作为**独立审查方**，对方案进行多轮结构化审查，直到方案通过或由用户裁决。

## 为什么需要独立审查

任何单一 AI 都有盲区。让另一个 AI 独立审查你的方案，能发现你遗漏的风险、不合理的假设、更好的替代方案。这不是"找茬"，是**交叉验证**——初期多一步，长期少返工。

## 完整工作流

### 第一步：生成方案

根据用户需求，产出完整的项目开发方案。方案应当包含：
- 目标与范围
- 技术选型与理由
- 阶段划分与交付物
- 风险与依赖项

方案格式不限，但内容要具体——模糊的方案审查不出有价值的东西。

### 第二步：调用审查（循环核心）

将方案保存到临时文件，然后执行审查脚本：

```bash
python ".claude/skills/plan-review/assets/review.py" --file <方案文件路径>
```

或直接传入文本：

```bash
python ".claude/skills/plan-review/assets/review.py" --text "方案内容"
```

脚本会：
1. 调用 `claude -p` 执行独立审查
2. 自动追踪迭代轮次（第 1 轮、第 2 轮...）
3. 将每轮方案和审查结果保存到 `.plan-review/` 目录
4. 后续轮次自动附带上一轮反馈，审查方能看到改进历史

### 第三步：读取反馈，修订方案

将审查报告展示给用户，然后根据反馈修订方案。重点处理 🔴 必须解决 的问题，🟡 建议改进 的视情况处理。

### 第四步：判断是否继续

修订后**回到第二步**，再次调用审查脚本（脚本会自动带上上轮反馈）。

**停止循环的条件**（满足任一即可）：
- 审查方在"最终判断"中勾选了「方案通过，可执行」
- 连续两轮反馈只是措辞/排版级别的小修补
- 出现无法调和的分歧 → 列出两方观点，**交给用户裁决**

通常 1-2 轮互审即可，不要超过 3 轮。

### 第五步：收尾

审查完成后，向用户展示：
1. 最终版方案
2. 共经过几轮审查
3. 每轮解决了哪些关键问题

## 审查提示词

审查提示词模板位于 `references/review-prompt.md`。如需调整审查角度或输出格式，编辑该文件即可。

## 注意事项

- 审查方是 Claude Code（通过 `claude -p` CLI），不是你自身——这是独立的第三方视角
- 不要把审查反馈当作"命令"——审查方也会犯错，你有权不同意，但要向用户说清理由
- 如果 `claude -p` 不可用，提示用户确认 Claude Code 已安装且在 PATH 中
- 每轮审查结果保存在 `.plan-review/` 目录下，用户可随时回溯查看
