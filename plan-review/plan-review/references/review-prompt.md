## 审查要求

请从以下四个维度逐一审查：

1. **完整性**：方案是否覆盖了所有必要方面？有没有遗漏的关键环节（如测试、部署、回滚、监控）？
2. **技术选型**：所选技术栈是否合理？有没有更简单或更成熟的替代方案？是否存在过度设计或技术风险？
3. **阶段划分**：里程碑和优先级是否合理？阶段之间的依赖关系是否清晰？有没有可以并行的工作被串行化了？
4. **风险与依赖**：外部依赖是否识别清楚？风险缓解措施是否到位？有没有隐含假设没有被显式声明？

```
## 问题清单

| 上轮问题 | 状态 | 说明 |
|---------|------|------|

## 亮点

- [方案中做得好的地方，1-3 条]

## 最终判断

- [ ] 方案通过，可执行
- [ ] 方案基本可行，建议改进上述问题后执行
- [ ] 方案存在结构性问题，需要重大修订
```

## 审查原则

- 直接指出问题，不需要客套
- 区分"必须解决"和"锦上添花"——不要把所有问题都标红
- 如果方案本身很好，大方说"方案通过"，不要为了显得有价值而强行找问题
- 给出的改进建议要具体可操作，不要说"建议进一步优化"这种废话
- 如果是后续轮次，重点关注上轮未解决的问题，已解决的不要重复提出
