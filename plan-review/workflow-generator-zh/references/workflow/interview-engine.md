# 采访引擎

## 目标

将用户的自然语言思路通过三层采访模型转化为结构化工作流参数。AI 始终先给出建议，然后用户确认或修改。

## 三层模型

### 第一层：领域定位

**目的：** 确认这个思路属于什么领域。

**规则：** 从用户的描述中推断领域标签、目的和用户角色。将推断作为建议呈现。在用户确认之前不要继续。

**模板：**
> 根据你的描述，这看起来是一个 **{domain_label}** 工作流。它的核心是 {domain_purpose}。是否正确？

**产出：** `generator-context/workflow/domain-analysis.md`

**红色警告——出现以下情况不得继续：**
- 用户描述过于模糊，无法推断任何领域
- 用户拒绝你的建议但没有提供替代方案
- 用户描述了多个不相关的领域

### 第二层：工作流结构

**目的：** 定义工作流骨架——输入、阶段、Skill 包名称、交付物、质量门禁。

**规则：** 对 5 个维度中的每一个，基于领域知识给出具体建议。每次呈现一个维度。不要跳过用户可能不关心的维度——建议默认值，让用户接受。

**五个维度（按此顺序呈现）：**

1. **输入材料**——将每个材料分类为 `required`（必需）、`strongly_recommended`（强烈推荐）或 `optional`（可选）。对每个材料，解释缺失的影响。

2. **工作流阶段**——建议 7-14 个有序步骤。每个步骤必须有清晰的输入和输出。最后两个步骤始终是 `quality_gates`（质量门禁）和 `delivery_report`（交付报告）。

3. **Skill 包名称**——生成的 Skill 包目录名，即交付侧目录。根据工作流用途建议（如 `test-coverage-analyzer`、`code-review-assistant`、`pipeline-runner`）。遵循小写连字符命名模式。注意：治理侧目录 `generator-context/` 是生成器自身使用的固定名称，不需要用户命名，也不应放入生成的 Skill 包内。

4. **核心交付物**——列出文件名和格式。文件名应具有描述性，不能是通用的（`需求覆盖率报告.xlsx` 而非 `report.xlsx`）。

5. **质量门禁**——每项必须可验证。包含验证方式：`script`（自动化）、`manual`（人工审查）、`format_check`（模式校验）。

**产出：** `generator-context/workflow/interview-answers.yaml`

**红色警告：**
- 用户对超过 2 个维度说"不知道"——建议具体示例
- 用户要求少于 5 个阶段——解释为什么需要分解
- 用户拒绝小写连字符命名模式——解释 Skill 包的命名规范

### 第三层：细节填充

**目的：** 填补第二层中发现的缺口——边界情况、错误处理、领域特定约束。

**规则：** 仅在实际存在缺口时触发。不要问第二层已经回答过的问题。最多 3 轮。

**触发条件：**
- 输入格式描述模糊
- 错误处理偏好不明确
- 缺少工具链依赖
- 约束条件冲突

**产出：** 合并到 `interview-answers.yaml`

**红色警告：**
- 问一个已经回答过的问题——先重新读取 interview-answers.yaml
- 超过 3 轮细节问题——升级到 stop_and_report
