# SKILL.md 生成规则

## 目标

将结构化采访答案（`interview-answers.yaml`）转化为完整的 8 段 SKILL.md。

## 生成过程

1. 加载 `interview-answers.yaml`
2. 使用 Jinja2 模板 `assets/templates/skill-md-template.md` 做第一轮填充
3. AI 润色：展开简洁描述、确保流程自然、添加领域特定的细微差别
4. 呈现给用户审查

## 各段推导规则

### 1. 运营模型

- `evidence_label`：从输入材料推导（如"需求文档和测试用例"→"需求追溯证据"）
- `delivery_label`：从核心交付物推导（如"覆盖率报告生成"）
- `governance_items`：标准集合——工作流状态、材料清单、阻断报告、用户决策
- `delivery_items`：来自采访答案中的 `core_deliverables`

### 2. 必需工作流

- 直接从采访答案中的 `workflow_phases` 获取
- 始终将 `stop_and_report` 说明作为最后一句追加
- 每个阶段从采访数据中获得一句话描述

### 3. 阶段 + 状态模型

- `phase` 值：`workflow_phases` 中的所有阶段名称
- `status` 值：标准 6 个（pending、in_progress、blocked、needs_review、done、deprecated）
- 包含阻断格式 YAML 示例
- 包含从阶段依赖关系推导出的合法回退表

### 4. 决策树

从阶段依赖分析推导：
- 如果某阶段需要材料 X → "如果 X 缺失，不要进入阶段 Y"
- 如果用户提供新材料 → "返回到阶段 Z"
- 优先级顺序：摄入检查 → 阻断检查 → 继续检查

### 5. 硬规则

反向推导自：
- 质量门禁：每个门禁 → 至少一条规则
- 阻断条件：每个阻断 → 一条阻止被阻断操作的规则
- 领域常识：来自工作台模式的标准规则（不要跳过日志、不要隐藏阻断等）

### 6. 资源映射

生成自：
- 每个工作流阶段 → 至少一个 references/ 条目
- `scripts/` 中的每个脚本 → 一个条目
- 标准治理条目（stop-and-report、state-management）

### 7. 质量门禁

直接来自采访答案中的 `quality_gates`，扩展标准工作台门禁：
- 所有配置文件存在且内部一致
- 每个主要声明在证据目录中有来源
- 交付物存在于输出目录中
- 工作流日志反映当前阶段和状态

### 8. 交付合约

来自采访答案中的 `core_deliverables` 和 `output_dir_name`。
格式：每个交付物独立一行，路径相对于输出目录。

## AI 润色检查清单

模板填充后，验证：
- [ ] 无模板语法残留（无 `{{ }}` 或 `{% %}`）
- [ ] 所有路径使用采访答案中正确的目录名
- [ ] 阶段描述是完整句子，不是片段
- [ ] 硬规则是领域特定的，不是通用的
- [ ] 决策树分支可操作（如果 X → 做 Y）
